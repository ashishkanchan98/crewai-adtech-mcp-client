What is the ReAct loop and why does every agent use it?ReAct Loop▼

ReAct (Reasoning + Acting) is the cognitive pattern each agent follows. Instead of answering in one shot, the agent alternates between thinking and doing:

  • Thought — LLM reasons: "I need to check the hourly spend curve first."
  • Action — calls an MCP tool: getHourlySpendCurve(campaignId="C123", days=3)
  • Observation — reads the tool's JSON return value
  • Reflection — updates understanding: "Budget hit at 10:14am — this is a pacing issue."

It loops until the task's expected_output is satisfied or max_iter is reached. More reliable than a one-shot prompt because the agent can self-correct mid-task.

What happens when the ReAct loop hits max_iter?ReAct Loop▼

CrewAI forces the LLM to produce a final answer with whatever data it has collected — even if incomplete. The downstream synthesis agent sees partial data and signals the limitation in its summary.

Keep max_iter between 5–10. Too low = incomplete analysis. Too high = slow responses and high token cost.
How does the Thought step know which MCP tool to call?ReAct Loop▼

Each agent gets a filtered tool subset via filter_tools(all_tools, "campaign"). The LLM sees tool names and their docstrings and decides which to call based on the task description. CrewAI wraps each MCP tool as a StructuredTool — the tool schema is injected into the system prompt automatically.

Why are all 7 agents instantiated at startup instead of per-request?Agents▼

Agent instantiation connects to the MCP server via SSE and loads all tool schemas — this adds 2–5 seconds if done per-request. CrewRouter builds the pool once at startup and reuses the same 7 agent objects. Only the tasks are created per-request, carrying the specific campaign_id and query context.

The shared pool is safe for concurrent requests because tasks are stateless — agents don't store per-request state.
Why is allow_delegation=False on all agents?Agents▼

When allow_delegation=True, agents can hand off subtasks to other agents mid-execution, causing unpredictable tool access and potential infinite delegation loops. Multi-agent collaboration is handled explicitly through the task context chain (context=[task1, task2]), which is deterministic and auditable.

What is MCP and why use SSE transport instead of HTTP REST?MCP Tools▼

MCP (Model Context Protocol) is an open standard for connecting LLM agents to tools. Instead of a custom REST client per tool, CrewAI discovers all 22 tools automatically from the MCP server's manifest. SSE provides a persistent connection — tool calls flow over a single long-lived HTTP stream, avoiding per-call TCP handshake overhead.

How does SSE work inside the MCP server — what actually happens on the wire?MCP Tools▼

SSE = Server-Sent Events — a one-way persistent HTTP connection where the server pushes data to the client over time. The connection stays open for the lifetime of the agent session.

How it works in this project:

CrewAI Client (port 8090)          MCP Server (port 8085)
        |                                   |
        |--- GET /sse (HTTP request) ------>|
        |                                   |
        |<-- HTTP 200 (connection stays open, text/event-stream)
        |                                   |
        |<-- event: session established ----|
        |                                   |
        |--- tool call: getCampaignStatus() |  (POST /messages/)
        |<-- event: tool result ------------|
        |                                   |
        |--- tool call: getHourlySpendCurve()|
        |<-- event: tool result ------------|
        |                                   |
        |  (connection stays open...)       |
        |--- tool call: getPacingSettings() |
        |<-- event: tool result ------------|

Why SSE instead of regular HTTP REST:

Regular REST SSE (MCP)
New TCP connection per tool call One persistent connection for all tool calls
Client pulls data Server pushes data as events
~100–300ms overhead per call Connection cost paid once at startup (~5–20ms per call)
Stateless Session-aware (session ID tracks conversation)

The 421 error on Cloud Run happened because FastMCP's TransportSecurityMiddleware validates the HTTP Host header on the GET /sse request. Cloud Run's load balancer sends its own hostname (adtech-mcp-server-xxxx.run.app) which wasn't in the allowlist ["127.0.0.1:*", "localhost:*"]. Fixed by checking the K_SERVICE env var (always set on Cloud Run) and disabling host validation only in that environment.

In server.py: mcp.sse_app() creates the full SSE ASGI app with both routes wired up. uvicorn.run(mcp.sse_app(), host="0.0.0.0", port=port) is all that's needed to serve all 22 tools over SSE.
Do the MCP tools return real data or mock data?MCP Tools▼

Currently realistic mock data — hardcoded JSON mirroring real DSP/SSP API responses. To connect real APIs, replace the function bodies in apis/campaign_api.py etc. The MCP tool interface (@mcp.tool() decorator) stays identical — agents won't know the difference.

Mock-first is intentional for the POC: validate agents against realistic data shapes before integrating production APIs.
Why did the MCP server return HTTP 421 on Cloud Run and how was it fixed?MCP Tools▼

FastMCP auto-enables DNS rebinding protection when starting on 127.0.0.1, validating the Host header against ["127.0.0.1:*", "localhost:*"]. Cloud Run sends its service URL as the Host header → not in allowlist → HTTP 421.

Fix: Cloud Run always injects K_SERVICE env var. When set → TransportSecuritySettings(enable_dns_rebinding_protection=False). When absent (local) → None so FastMCP keeps localhost protection.

Always rebuild (gcloud builds submit) before redeploying — deploy only re-runs the existing image.
How does the keyword router classify queries — what if it picks the wrong use case?Crew & Routing▼

Scores each of 7 use cases by counting keyword hits in the query. Highest score wins. Zero hits → defaults to delivery_failure. Ties → first in dict order.

For production, replace with an LLM intent classifier (one cheap groq/llama-3.1-8b-instant call) that understands full query context — not just keyword counts.
What is the difference between sequential tasks and UC6's parallel tasks?Crew & Routing▼

Sequential (UC1–5, UC7): Tasks run one after another, each reading prior outputs via context=[prev_task]. Parallel (UC6): All 6 audit tasks have async_execution=True — CrewAI fires them simultaneously. The GO/NO-GO synthesis task waits for all 6. Running in parallel cuts wall-clock time from ~6× to ~1× the slowest single audit.

The overall Process is still sequential — async_execution=True is a per-task flag. The synthesis task always runs last.
How does task context chaining work?Crew & Routing▼

Each Task has context=[] listing earlier tasks whose text outputs are injected into its LLM prompt. The synthesis task in UC1 has context=[triage, campaign_check, audience_check, deal_check] — the LLM sees all 4 agents' findings and produces a ranked action plan citing specific values from each.

Context is append-only — earlier agents cannot see later agents' outputs. Data flows strictly forward.
Why two separate Cloud Run services instead of one?Deployment▼
  • Independent scaling — MCP server scales to 0 when idle; client can stay warm
  • Independent deployment — update MCP tools without redeploying the client
  • Protocol separation — MCP uses SSE (long-lived connections); FastAPI uses standard HTTP. Mixing complicates Cloud Run timeout settings
  • Security boundary — each service gets its own IAM role
How do I stop billing on Cloud Run?Deployment▼

Cloud Run charges only when processing requests. With --min-instances=0 (default) the service scales to zero when idle — no charge.

  • Delete: gcloud run services delete adtech-mcp-server --region asia-south1 --project adtech-support-poc
  • Or scale to zero: gcloud run services update adtech-mcp-server --min-instances 0 --region asia-south1

Artifact Registry charges ~$0.10/GB/month for image storage. Delete images with gcloud artifacts docker images delete ... to eliminate storage costs.

Is keyword routing production-ready?Design Decisions▼

Not production-ready. Breaks when users don't use domain keywords, queries span multiple domains, or new terminology appears. Production alternatives: LLM intent classifier (cheapest), embedding similarity, or multi-crew parallel (most accurate).

How do I add a new use case (UC8)?Design Decisions▼

4 file changes only:

  • router.py — add entry to USE_CASE_META with label + keywords
  • tasks.py — add uc8_my_use_case_tasks(agents, ctx) returning list[Task]
  • crews.py — add task factory to _TASK_FACTORIES and agent keys to _USE_CASE_AGENTS
  • server.py — add any new @mcp.tool() functions needed

No changes to FastAPI routes, agent pool, or UI — the system is fully data-driven.

What happens if the MCP server is unreachable at startup?Design Decisions▼

FastAPI logs a warning but does not crash. mcp_tools is set to an empty list. The CrewRouter initializes with zero MCP tools — agents only have the KB tool. Queries still route and agents still run, but MCP tool calls return "tool not available" errors.

Start MCP server first, then the FastAPI client to avoid this state during development.
Why Groq llama-3.3-70b-versatile instead of GPT-4o or Gemini?LLM & Models▼

Three reasons: speed, cost, and zero lock-in.

  • Speed: Groq's LPU hardware runs llama-3.3-70b at ~200 tokens/sec vs OpenAI's ~40 tokens/sec. A 7-agent crew generating ~14,000 reasoning tokens completes in ~70s on Groq vs ~5 min on GPT-4o.
  • Cost: Groq llama-3.3-70b ≈ $0.59/1M input tokens vs GPT-4o $2.50/1M — 4× cheaper for the same model size class.
  • Zero lock-in: llm_provider=vertex in .env switches the entire system to Gemini 2.0 Flash on Vertex AI — no code changes, same agent/task definitions, CrewAI's litellm bridge handles the translation.
Groq is the default for dev/POC. Set LLM_PROVIDER=vertex for GCP production — Gemini Flash is multimodal (native image analysis) and has no cold-start latency on Cloud Run since it's in the same GCP project.
Why temperature=0.2 — not 0.0 for full determinism or 0.7 for creativity?LLM & Models▼
TemperatureBehaviorRisk for support agents
0.0Fully deterministic — identical output every runRigidly literal — won't rephrase a failed tool call, causing stuck loops

How does MCPServerAdapter auto-discover all 22 MCP tools at startup?MCP Tools▼

Calling adapter.__enter__() in connect_mcp_adapter() triggers the full MCP protocol handshake:

FastAPI startup
    │
    ▼
connect_mcp_adapter("http://localhost:8085/sse")
    │
    ▼
MCPServerAdapter.__enter__()
    ├── GET /sse  ──────────────────▶  MCP Server
    │              persistent SSE connection established
    │
    ├── send: {"method": "initialize"}
    │              ◀── response: server capabilities
    │
    ├── send: {"method": "tools/list"}
    │              ◀── response: [{name, description, inputSchema}, ×22]
    │
    └── converts each JSON Schema → CrewAI BaseTool subclass
            └── returns list[BaseTool]  (22 tools)

Each tool's inputSchema is passed to pydantic to auto-generate argument validators. This is why agent tool calls are type-safe — if the LLM passes a string where an integer is expected, pydantic raises a validation error before the MCP call is even sent.

filter_tools(all_tools, "campaign") runs after discovery — it slices the full 22-tool list to the 5 tools assigned to the Campaign Analyst. The LLM only sees those 5 tool names in its system prompt.
What happens when an MCP tool call fails mid-ReAct loop — timeout, wrong args, or server error?MCP Tools▼

CrewAI catches the tool exception and converts it into an Observation the LLM can reason about:

Thought: I need campaign status for C123.
Action: getCampaignStatus(campaign_id="C123")
Observation: Error — connection timeout after 30s

Thought: The tool timed out. I'll try getHourlySpendCurve instead
         to infer delivery from the spend pattern.
Action: getHourlySpendCurve(campaign_id="C123", days=1)
Observation: {"hourly": [{"hour": 0, "spend": 0}, ...]}

Three failure modes and their outcomes:

  • Wrong args (validation error): pydantic catches it before the MCP call — LLM sees the schema error and retries with corrected arguments
  • Timeout / server error: SSE error event → Observation. LLM pivots to an alternative tool or proceeds with partial data
  • Repeated failure: after max_iter loops, CrewAI forces a final answer — downstream synthesis task sees "campaign status unavailable" and flags it in the report
Set CREW_VERBOSE=true in .env to see every Thought / Action / Observation in FastAPI logs — the single most useful debugging tool for stuck ReAct loops.

What is RAG and how is it implemented in this project?RAG▼

RAG (Retrieval-Augmented Generation) grounds LLM responses in factual documents rather than relying solely on model training data. Instead of trusting the LLM to recall troubleshooting guides from memory, the agent searches a vector database and injects the relevant runbook as context into the prompt.

In this project, RAG is implemented as a CrewAI tool (KBSearchTool) rather than a middleware layer. The Triage agent calls searchKnowledgeBase() as its first ReAct action, retrieves the top 3 matching runbooks from ChromaDB, then uses that content to inform its analysis and the downstream agents' context.

Tool-wrapped RAG is a key design choice: the agent autonomously decides when to search (based on task description), not on every request unconditionally. This avoids irrelevant KB results in use cases like UC3 (IVT/Fraud) where the Triage agent is not in the task chain.

Is this RAG implementation production-ready — what are the critical gaps?RAG▼

The current implementation is a solid POC but has four critical gaps before it is production-safe:

  • No score threshold: ChromaDB always returns top-3 results, even when the best match has a cosine score of 0.1 (near-random). The LLM receives garbage context and produces confidently wrong answers. Fix: filter out results below 0.5 in _kb_search_local() and return a "no relevant KB article found" signal instead.
  • No hybrid search: Semantic (dense) search alone misses exact product names and acronyms that BM25 (keyword) catches trivially. BM25 alone misses synonyms. Production RAG always merges both via Reciprocal Rank Fusion (RRF).
  • ChromaDB is single-node: It uses a file-locked SQLite backend. Fine for a single Cloud Run instance, but breaks under horizontal scaling. For multi-instance production, replace with Qdrant, pgvector, or just set KB_PROVIDER=gcp.
  • No re-ranker: Cosine similarity is a biased proxy for relevance. A cross-encoder re-ranker compares query and document together (not as independent vectors) and produces much more accurate relevance scores. It is the single highest-impact upgrade for retrieval precision.
Score threshold and re-ranker together take about 1 day to add and eliminate the most dangerous failure mode: the system confidently citing an irrelevant runbook as if it were authoritative.
What is hybrid search and why does pure semantic search fall short?RAG▼

Hybrid search combines two fundamentally different retrieval signals and merges their ranked lists:

Dense (Semantic)Sparse (BM25)
MethodVector cosine similarityTerm frequency / inverse doc frequency
StrengthSynonyms, paraphrases, conceptual meaningExact match — product names, IDs, acronyms
WeaknessMisses "getCampaignStatus", exact error codesMisses "bot traffic" → "IVT" synonym
AdTech example"my ads stopped" → finds delivery runbook ✓"deal ID 5813 zero bids" → finds PMP doc ✓

Reciprocal Rank Fusion (RRF) merges the two ranked lists without needing to normalize scores (they're on different scales). Each doc's merged score = Σ 1/(k + rank_i) where k=60. The top results of the merged list beat either signal alone.

In this project, add BM25 with the rank_bm25 library (pure Python, no new services). Build a BM25 index over the same kb-docs/ JSON files, run both searches, merge with RRF, then pass the merged top-10 to the re-ranker.

What is RAGAS and how do you evaluate RAG quality without ground truth?RAG▼

RAGAS (Retrieval-Augmented Generation Assessment) is an evaluation framework that measures RAG quality without requiring human-labeled ground truth for every query. It uses the LLM itself as a judge across four dimensions:

  • context_recall (0–1): Of all the information needed to answer the question, how much is present in the retrieved docs? Low score = KB missing content or retrieval failing to find it.
  • faithfulness (0–1): Does the generated answer only contain claims supported by the retrieved context? Low score = LLM hallucinating beyond the KB.
  • answer_relevancy (0–1): Does the generated answer actually address the question asked? Can be low even with high faithfulness if the retrieved docs were off-topic.
  • context_precision (0–1): Are the retrieved docs actually relevant, or are there noisy irrelevant ones in the top-3? Low score = re-ranker or threshold needed.

Minimal setup for this project:

pip install ragas

# Build a 20-query golden set: question + ground_truth_answer

            "ground_truth": "Safari ITP 2.x limits cookie lifetime..."}]

results = evaluate(dataset, metrics=[faithfulness, answer_relevancy, context_recall])
# Returns a DataFrame with per-question scores
Target thresholds for production: context_recall > 0.80, faithfulness > 0.85, answer_relevancy > 0.80. Wire RAGAS into CI: fail the deploy if any metric drops below threshold on the golden test set.

# Reciprocal Rank Fusion — merge dense and sparse results
def rrf_merge(dense_results, sparse_results, k=60):
    scores = {}
    for rank, doc_id in enumerate(dense_results):
        scores[doc_id] = scores.get(doc_id, 0) + 1 / (k + rank + 1)
    for rank, doc_id in enumerate(sparse_results):
        scores[doc_id] = scores.get(doc_id, 0) + 1 / (k + rank + 1)
    return sorted(scores, key=scores.get, reverse=True)

A dense vector is a compact, fixed-size array where every slot contains a floating-point value — no zeros, no empty positions. A neural network (SentenceTransformers, OpenAI embeddings, BGE) reads the full text and compresses its meaning into a small array. This project uses all-MiniLM-L6-v2 which outputs 384 dimensions.

What is SPLADE — how is it different from BM25 and dense vectors?RAG▼

Classic BM25 counts only exact words; SPLADE "guesses" likely synonyms and adds them to the sparse vector with learned weights.

  → "impressions": 0.29  ← learned context            ✅
  → Still sparse: 99%+ of 30,000 vocabulary slots = 0

What is LangGraph — how is it different from LangChain and CrewAI?LangGraph▼

LangGraph is a library (part of the LangChain ecosystem) for building stateful, graph-based agent workflows. Where LangChain gives you linear chains and CrewAI gives you sequential/parallel task crews, LangGraph gives you a directed graph of nodes where each node is an agent step and edges encode conditional branching logic.

Core Concept — State Machine over a Graph

LangGraph workflow = nodes + edges + shared state

  Node     = a function or LLM call that reads/writes shared state
  Edge     = transition between nodes (unconditional or conditional)
  State    = TypedDict shared across all nodes (like a blackboard)
  Graph    = StateGraph compiled into a runnable

  Example: AdTech diagnosis workflow as a LangGraph

  State = {query, campaign_id, triage_result, deal_result, final_report}

  Nodes:
    triage_node    → calls LLM with KB context, writes triage_result
    deal_node      → calls deal check tool, writes deal_result
    audience_node  → calls audience check tool
    synthesis_node → reads all results, writes final_report
    end_node       → returns final_report to caller

  Edges:
    START → triage_node
    triage_node → deal_node          (always)
    triage_node → audience_node      (always, parallel branch)
    deal_node   → synthesis_node
    audience_node → synthesis_node
    synthesis_node → END

  Conditional edge example:
    triage_node → ivt_node    IF triage_result contains "fraud"
    triage_node → deal_node   OTHERWISE

What is LCEL (LangChain Expression Language) and how does it make chains composable?LangChain▼

LCEL is LangChain's pipe-operator syntax for composing chains. Any object that implements Runnable (LLMs, prompts, retrievers, parsers, tools) can be chained with | — the output of the left becomes the input of the right. The result is itself a Runnable that supports streaming, async, batching, and retry out of the box.

# Classic approach (verbose):
prompt_value = prompt.format(query=query)
llm_output = llm.invoke(prompt_value)
result = parser.parse(llm_output)

# LCEL — same thing, one line:
chain = prompt | llm | StrOutputParser()
result = chain.invoke({"query": "why are bids zero?"})

# RAG chain with retriever:
rag_chain = (
    {"context": retriever, "question": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)
answer = rag_chain.invoke("deal D456 zero bids")

# Streaming (works automatically on any LCEL chain):
for chunk in rag_chain.stream("deal D456 zero bids"):
    print(chunk, end="", flush=True)

# Async batching:
answers = await rag_chain.abatch(["query1", "query2", "query3"])

Full hybrid RAG pipeline as one LCEL chain:

from langchain_community.retrievers import BM25Retriever

Classic RAG vs Graph RAG vs Agentic RAG — how do the three architectures differ and when does each win? — how do the three architectures differ and when does each win?RAG▼

Three generations of RAG architecture, each solving a failure mode of the previous one. Classic RAG is the baseline; Graph RAG adds relational reasoning; Agentic RAG adds multi-step planning. They are not mutually exclusive — production systems often combine all three.

Classic RAG — The Baseline

User query
  → embed query → vector search → top-k chunks → inject into prompt → LLM answer

Knowledge structure: flat chunk pool (no relationships between chunks)
Retrieval unit:      fixed-size text chunk (paragraph, page, sentence)
Retrieval logic:     one-shot cosine similarity (or + BM25 hybrid)
Reasoning:           entirely inside the LLM prompt — one inference call

Example query it handles well:
  "What causes zero bids on a PMP deal?"
  → retrieves the PMP runbook chunk → LLM answers directly

Example query it fails on:
  "Which deals share the same floor price issue as deal D456?"
  → retrieves D456 chunk only — doesn't know D789 and D012 are related
  → answer is incomplete; the relationship lives between documents, not inside one

Graph RAG — Relational Knowledge

Builds a knowledge graph on top of (or instead of) a flat vector index. Entities (deals, campaigns, seats, buyers) are nodes; relationships (shares_floor_price_with, blocked_by, owned_by) are edges. Retrieval traverses the graph — not just similarity.

User query
  → entity extraction ("D456", "floor price")
  → graph traversal: D456 → [shares_floor_with] → D789, D012
  → fetch node summaries + community summaries
  → inject into prompt → LLM answer

Knowledge structure: entity graph (nodes + typed edges + community clusters)
Retrieval unit:      entity + its neighborhood (1–2 hops)
Retrieval logic:     graph traversal + optional vector search on node embeddings
Reasoning:           graph walk is deterministic; LLM synthesizes over retrieved subgraph

Example query it handles well (Classic RAG fails):
  "Which deals share the same floor price issue as deal D456?"
  → traverses D456 → floor_price_group → [D456, D789, D012]
  → answers with all three deals + shared root cause

Graph RAG implementations:
  Microsoft GraphRAG  — community detection (Leiden algorithm), global/local search modes
  Neo4j + LangChain   — Cypher query generation from natural language
  LlamaIndex          — KnowledgeGraphIndex with NetworkX backend

Agentic RAG — Multi-Step Planning

The LLM plans its own retrieval strategy. Instead of one fixed retrieve → answer step, an agent iterates: searches, reads results, decides what else it needs, searches again, synthesizes. This project uses Agentic RAG — the Triage agent autonomously calls searchKnowledgeBase() as an Action in its ReAct loop.

User query
  → Agent Thought: "I need to check the deal runbook and the seat eligibility guide"
  → Action 1: searchKnowledgeBase("PMP deal zero bids")        ← retrieval #1
  → Observation: [PMP runbook, deal checklist]
  → Agent Thought: "Floor price looks fine. Need to check seat eligibility."
  → Action 2: searchKnowledgeBase("seat eligibility SSP block") ← retrieval #2
  → Observation: [seat eligibility runbook]
  → Agent Thought: "Both found. Ready to synthesize."
  → Final Answer: ranked action plan citing both runbooks

Knowledge structure: same as Classic RAG (flat chunks), but agent decides what to retrieve
Retrieval unit:      agent-chosen query per iteration
Retrieval logic:     agent formulates queries based on intermediate findings
Reasoning:           multi-step; agent self-corrects between retrievals

Key distinction from Classic RAG:
  Classic RAG: one query → one retrieval → one answer
  Agentic RAG: one query → N retrievals → synthesis (N decided at runtime)

RAG vs CAG — what is Cache-Augmented Generation and when should you use one over the other?RAG▼

RAG (Retrieval-Augmented Generation) retrieves relevant document chunks at query time and injects them into the prompt. CAG (Cache-Augmented Generation) skips retrieval entirely — it preloads the entire knowledge base into the LLM's KV cache once, then answers every query from that cached context.

Architecture Comparison

RAG — retrieval at query time:
  User query
    → embed query (MiniLM, ~50ms)
    → vector search ChromaDB (top-3 chunks)
    → inject chunks into prompt
    → LLM generates answer

  Pipeline: Query → Embed → Retrieve → Augment → Generate

CAG — preloaded KV cache:
  Startup (once):
    → load ALL KB documents into LLM context window
    → LLM computes KV cache for the full KB
    → cache stored in GPU memory

  User query
    → append query to cached context
    → LLM generates answer (cache already warm)

  Pipeline: Query → Generate  (retrieval step is gone)

The KV Cache — What CAG Actually Exploits

Modern LLMs (Gemini 1.5 Pro: 1M tokens, GPT-4o: 128K, Claude: 200K) can store their attention computation (Keys + Values) for a fixed prefix. If your entire KB is that prefix, every new query appends to it without re-computing the KB's attention — only the new query tokens are processed fresh.

KV cache hit (CAG):
  [  KB prefix — cached, not recomputed  ] [ new query tokens ]
  ←————————————————————————————————————→  ←————————————————→
         already in GPU memory                 computed now
  Cost: O(query_tokens)  ← cheap

No cache (RAG with injected chunks):
  [ system prompt ] [ retrieved chunk 1 ] [ chunk 2 ] [ chunk 3 ] [ query ]
  ←————————————————————————————————————————————————————————————————————————→
                    all recomputed every request
  Cost: O(prompt_tokens + chunk_tokens + query_tokens)  ← more expensive per call

Use RAG when:
  ✅ KB is large (thousands of documents, millions of tokens)
  ✅ KB updates frequently (new runbooks added daily)

  ✅ You need zero retrieval errors (medical, legal — missing one doc is unacceptable)
  ✅ You can afford a long-context API (Gemini 1.5 Pro at 1M tokens)
  ✅ Simplicity beats cost — no vector DB, no embeddings, no retrieval pipeline
  ✅ Latency is critical and KB is static (cache warms once, serves forever)

How does Reciprocal Rank Fusion (RRF) merge two ranked lists without normalizing scores — step-by-step with worked example?RAG▼

RRF merges ranked lists without needing to normalize scores — different retrieval systems produce scores on completely incompatible scales, so RRF ignores scores entirely and uses rank positions instead. Each doc's merged score = Σ 1/(k + rank_i) where k=60.

The Problem RRF Solves

Dense (ChromaDB cosine similarity):     Sparse (BM25 term frequency):
  Doc_A  score: 0.91                      Doc_D  score: 28.5
  Doc_B  score: 0.87                      Doc_E  score: 12.1
  Doc_C  score: 0.72                      Doc_A  score:  8.3
  Doc_D  score: 0.43                      Doc_B  score:  6.1
  Doc_E  score: 0.38                      Doc_C  score:  2.2

Can you add 0.43 + 28.5 to merge them?  NO.
BM25's 28.5 would drown out everything from dense completely.

RRF's insight: ignore the scores entirely, use only the rank positions.
Rank 1, rank 2, rank 3 — always on the same scale regardless of which system produced them.

The Formula — Step by Step

merged_score(doc) = Σ  1 / (k + rank_i)

Where:
  rank_i = position of this doc in list i  (1 = first, 2 = second, ...)
  k      = smoothing constant (default 60)
  Σ      = sum across every list the doc appears in

Worked Example — Query: "deal D456 zero bids"

Dense search results:             Sparse / BM25 results:
  rank 1 → Doc_A  (PMP runbook)    rank 1 → Doc_D  ← "D456" exact match
  rank 2 → Doc_B  (deal guide)     rank 2 → Doc_E  ← "seat" exact match
  rank 3 → Doc_C  (zero bids)      rank 3 → Doc_A
  rank 4 → Doc_D  ← our target     rank 4 → Doc_B
  rank 5 → Doc_E                   rank 5 → Doc_C

Apply score = 1/(60 + rank_dense) + 1/(60 + rank_sparse):

Doc_A:  1/(60+1) + 1/(60+3) = 1/61  + 1/63  = 0.01639 + 0.01587 = 0.03226
Doc_B:  1/(60+2) + 1/(60+4) = 1/62  + 1/64  = 0.01613 + 0.01563 = 0.03176
Doc_C:  1/(60+3) + 1/(60+5) = 1/63  + 1/65  = 0.01587 + 0.01538 = 0.03125
Doc_D:  1/(60+4) + 1/(60+1) = 1/64  + 1/61  = 0.01563 + 0.01639 = 0.03202  ← moves up
Doc_E:  1/(60+5) + 1/(60+2) = 1/65  + 1/62  = 0.01538 + 0.01613 = 0.03151

Sorted merged results:
  Rank 1 → Doc_A  0.03226   (was #1 dense,  #3 sparse)
  Rank 2 → Doc_D  0.03202   (was #4 dense,  #1 sparse) ← PROMOTED — exact D456 match
  Rank 3 → Doc_B  0.03176   (was #2 dense,  #4 sparse)
  Rank 4 → Doc_E  0.03151   (was #5 dense,  #2 sparse)
  Rank 5 → Doc_C  0.03125   (was #3 dense,  #5 sparse)

Doc_D was buried at rank 4 in dense-only search. BM25 found it at rank 1 (exact D456 match). RRF promotes it to rank 2. The LLM now receives the exact-match document it needs.

What k=60 Does

k is a smoothing constant that controls how steep the score drop-off is between ranks.

Without k (k=0):         With k=60 (default):
  rank 1 → 1/1  = 1.000   rank 1 → 1/61 = 0.01639
  rank 2 → 1/2  = 0.500   rank 2 → 1/62 = 0.01613  ← barely less than rank 1
  rank 3 → 1/3  = 0.333   rank 3 → 1/63 = 0.01587
  rank 10→ 1/10 = 0.100   rank 10→ 1/70 = 0.01429

k=0:   rank 1 is 2× better than rank 2  → winner-takes-all, aggressive
k=60:  rank 1 is barely better than rank 2  → gentle, balanced

k=60 means: "being first is good, but being second isn't a disaster." The value 60 came from the original RRF paper (Cormack et al., 2009) — tested empirically across many retrieval benchmarks. It became the standard default.

Why "Top Results Beat Either Signal Alone"

Dense-only top 3:   Doc_A, Doc_B, Doc_C     ← Doc_D (exact match) is missing
Sparse-only top 3:  Doc_D, Doc_E, Doc_A     ← Doc_C (semantic match) may be missing
RRF merged top 3:   Doc_A, Doc_D, Doc_B     ← contains BOTH the semantic AND exact match

A document that ranks highly in both lists gets contributions from both sides — it dominates. A document that only ranks high in one list still gets promoted, but less so. Documents that don't appear in either list get a score of zero. The merged top-3 combines orthogonal signals rather than betting everything on one.

One-line intuition: RRF says "I don't care how confident each system is — I only care about the ORDER each system produced. If both systems agree a doc is near the top → it wins."
